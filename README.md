# Evaluations - Directory Structure

This directory contains evaluation results for state machine diagram generation using different prompting strategies (single-stage vs. two-stage), along with human and automatic assessment via LLMs.

**Claude Sonnet 4.5** is the primary generator: its outputs were assessed by two human raters and serve as the reference for generation quality (RQ1) and the human-validated baseline for grading quality (RQ2). All three models — **Claude Sonnet 4.5**, **GPT-5.5**, and **Gemini 3.1 Pro Preview** — act as automated graders benchmarked against human assessment in RQ2. GPT-5.5 and Gemini 3.1 Pro Preview also generated additional state machines used to examine whether the RQ2 grading trends hold across different inputs (RQ3). Because these GPT-5.5- and Gemini-generated state machines were not assessed by human graders, they are used for grading-trend comparisons only, not for measuring generation quality.

| Paper RQ | What is evaluated | Evidence available |
|----------|-------------------|--------------------|
| **RQ1** | Generation quality of Claude Sonnet 4.5 using single-stage and two-stage prompting | Human assessment of Claude-generated state machines |
| **RQ2** | Grading quality of Claude Sonnet 4.5, GPT-5.5, and Gemini 3.1 Pro Preview as automated assessors | Each LLM grader compared against reconciled human grades for Claude-generated state machines (two-stage, 6-shot) using precision, recall, F1, confusion matrices, and weighted Cohen's κ |
| **RQ3** | Whether the assessment patterns from RQ2 hold for state machines generated by GPT-5.5 and Gemini 3.1 Pro Preview | Automatic grades from all three LLM graders on GPT-5.5- and Gemini-generated state machines; interpreted as grading-trend comparisons, not generation quality |

For RQ1, precision is very high for both configurations; recall and F1 distinguish single-stage from two-stage performance. For RQ2, weighted Cohen's κ is the primary agreement metric alongside per-element confusion matrices. For RQ3, Score 1 rates and partial-credit rates are compared across generator conditions without human ground truth.

> [!WARNING]
> **Known bug — Metrics sheets inside all Excel files**
> The Precision and Recall values in each workbook's Metrics sheet are incorrect. See the [Performance Metrics Explanation](#performance-metrics-explanation) section for details.

The repositories used to generate these results are:

- https://github.com/reheant/mermaid-parser-py — Mermaid parsing logic  
- https://github.com/reheant/StateMachineLLM — Generation, prompting, and visualization logic

## Examples

### Main Evaluation Systems (Non-Blind Examples — 1-stage & 2-stage, 3-shot and 6-shot)

3. **Dishwasher** - State machine for a dishwashing appliance
4. **Printer** - State machine for a printer device
5. **Spa Manager** - State machine for a spa management system

7. **Train Automation System** - State machine for train automation

8. **SSC7** - State machine for a Self-Service Checkout
9. **Wumple** - State machine for a watch called W-UMPLE

> SSC7 and Wumple (W-UMPLE) were used as blind test cases for the two-stage 6-shot configuration — excluded from all prompting and intermediate evaluation to assess generalization. They were not evaluated under the single-stage or two-stage 3-shot configurations.

## Directory Organization

### Reference Files (at Example Root Level)

#### `[Example Name] - Sample Solution.pdf`
**Purpose:** Contains the official solution or reference state machine diagram for the project

**Content:** Shows the expected state machine with correct states, transitions, events, guards, actions, and other elements

**Usage:** Use this as the ground truth when analyzing results

#### `[Example Name] - Description.pdf`
**Purpose:** Contains the complete problem description and requirements for the state machine. This was fed to the LLM to generate the evaluated state machines.  

**Content:** Detailed specifications for system behaviour, constraints, and expected functionality  

**Usage:** Reference this to understand what the LLM was asked to generate and what the grading criteria are based on 

#### `[Example Name] Evaluation - Template.xlsx`
**Purpose:** Evaluation template and grading rubric used for this project  

**Content:** Spreadsheet where elements from the sample solution are broken down into model elements, along with inter-rater calculations (Cohen's kappa and weighted Cohen's kappa) and evaluation metrics such as F1 score, precision, and recall  

**Usage:** Defines the evaluation framework and metrics used across all evaluation runs

#### `[example_name]_ground_truth_mermaid.txt`
**Purpose:** Ground truth state machine represented in Mermaid syntax, manually authored from the official sample solution

**Content:** Complete Mermaid `stateDiagram-v2` code capturing all states, transitions, guards, actions, composite states, and history states from the reference solution

**Usage:** Provided directly to the LLM grader as input during automatic grading, and paired with the description for 3-shot and 6-shot configuration during generation to give it an unambiguous, machine-readable representation of the expected state machine.

#### `[example_name]_ground_truth_mermaid_compiled.png`
**Purpose:** Rendered visualization of the ground truth Mermaid diagram

**Content:** PNG image produced by compiling `[example_name]_ground_truth_mermaid.txt` with the project's Mermaid parser and renderer tool

**Usage:** Allows quick visual inspection of the ground truth without using the Mermaid parser and renderer tool; useful for cross-checking that the Mermaid source parses correctly  

### Stage Folders

#### `1 stage/` - Single-Stage Baseline
Contains evaluation results from the single-stage baseline, where the LLM produces the complete UML state machine in a single prompt using a 4-block structure: role, syntax rules, in-context examples (3-shot), and a 7-step chain-of-thought task.

#### `2 stage/` - Two-Stage Refinement
Contains evaluation results from the two-stage refinement approach. The first stage is identical to the single-stage baseline (with potentially 3 additional in-context examples for the 6-shot configuration). The second stage passes the first-stage output into a refinement prompt that targets six specific error categories: missing history states, missing transitions, incomplete transition labels (absent guards or actions), collapsed multi-step sequences, flat structures where composite states are implied, and history state naming violations.

### Example Count Folders
Each stage sub-directory is named after the number of few-shot examples that
were supplied to the LLM during that evaluation run:

| Folder name   | Stage(s)            | Few-shot examples | Benchmark problems evaluated |
|---------------|---------------------|-------------------|------------------------------|
| `3-examples`  | 1 stage and 2 stage | 3                 | 7 (main examples only)        |
| `6-examples`  | 2 stage only        | 6                 | 9 (7 main + SSC7 + Wumple)    |

**`3-examples`** — the prompt included 3 few-shot examples. All single-stage evaluations and the earlier two-stage run used this configuration, evaluated on the 7 non-blind benchmark systems. The 3 in-context examples were drawn from a fixed pool of 4 (Printer, Spa Manager, Dishwasher, Chess Clock), always selecting the first three unless the system being evaluated was among them, in which case it was skipped and the fourth was substituted.

**`6-examples`** — the prompt was extended to 6 in-context examples, giving the model more demonstrations. This is the strongest two-stage configuration and the basis for RQ2 and RQ3. The 6 examples were drawn from a pool of 7 (all non-blind systems), always selecting the first six unless the system being evaluated was among them, in which case it was skipped and the seventh was substituted. SSC7 and W-UMPLE were evaluated only under this configuration, as blind test cases never used as shot examples in any run.

The example count is also embedded in each file name (e.g., `_3-examples_` vs
`_6-examples_`) so that individual files remain self-descriptive regardless of
which folder they reside in.

## Pipeline Script

### `run_pipeline.py`

**Purpose:** Runs the full evaluation pipeline in dependency order. Combines all compute, check, and figure-generation steps into a single entry point.

**Pipeline phases:**

| Phase | Scripts | Description |
|-------|---------|-------------|
| **data** | `compute_rq1_generation_quality`, `compute_rq2_grading_quality`, `compute_rq2_grading_results`, `compute_rq3_grader_stability`, `compute_element_distribution`, `compute_rq2_weighted_kappa`, `compute_rq2_human_grading_agreement` | Compute all CSV outputs |
| **checks** | `check_cohens_kappa_consistency`, `check_interrater_consistency` | Validate data integrity |
| **figures** | `generate_agreement_figures`, `generate_paper_rq2_confusion_matrices` | Generate all PNG figures |

**Usage:**
```bash
# Run all phases
python _scripts/run_pipeline.py

# Run a specific phase only
python _scripts/run_pipeline.py --phase data
python _scripts/run_pipeline.py --phase checks
python _scripts/run_pipeline.py --phase figures

# Skip figure generation
python _scripts/run_pipeline.py --skip-figures
```

## Scripts

### `_scripts/compute/`

#### `_xlsx_utils.py`

**Purpose:** Internal shared module with utility functions for reading Excel (xlsx) workbooks. Imported by the compute scripts; not intended to be run directly.

#### `compute_rq1_generation_quality.py`

**Purpose:** Reads the human-grading section (top of the `Metrics` sheet, rows 3–10) from every `*CombinedHumanGradingVsClaude4.5SonnetAutoGrading_Claude4.5SonnetGeneration.xlsx` workbook across all stages and shot configurations. Pools raw N / TP / FP / FN counts across all examples per (approach, element type), derives an Overall row by summing the element-type pools, and computes micro-averaged Precision / Recall / F1 (rounded to 3 decimal places) from the pooled totals. Element types are output in alphabetical order.

**Usage:**
```bash
python _scripts/compute/compute_rq1_generation_quality.py
python _scripts/compute/compute_rq1_generation_quality.py --out-dir /custom/path
python _scripts/compute/compute_rq1_generation_quality.py --print   # stdout only
```

**Outputs** (written to `_Figures/data/rq1/`):
- `rq1_generation_quality_summary.csv` — pooled N/S/TP/FP/FN counts and micro-averaged Precision/Recall/F1 per approach and element type (alphabetical), with a derived Overall row per approach (S = weighted true positives; see [Performance Metrics Explanation](#performance-metrics-explanation))
- `rq1_generation_quality_by_example.csv` — one row per (approach, project, element type) with raw N/S/TP/FP/FN counts and computed Precision/Recall/F1 (3 d.p.); an additional `model_element = "Overall"` row per (approach, project) aggregates across all element types; for `two-stage (6 examples)` only, two extra rows with `project = "main examples"` and `project = "blind test"` further aggregate the 7 non-blind systems and SSC7 + Wumple respectively

#### `compute_rq2_grading_quality.py`

**Purpose:** Reads the `Weighted Cohens Kappa` sheet from every 2-stage/6-examples `CombinedHumanGradingVs*AutoGrading*.xlsx` workbook and builds per-file confusion matrix counts for RQ2.

**Usage:**
```bash
python _scripts/compute/compute_rq2_grading_quality.py
python _scripts/compute/compute_rq2_grading_quality.py --out-dir /custom/path
python _scripts/compute/compute_rq2_grading_quality.py --print
```

**Output** (written to `_Figures/data/rq2/`):
- `rq2_confusion_matrices.csv` — raw confusion counts (human_score, llm_score) by grader and element type

#### `compute_rq2_grading_results.py`

**Purpose:** Computes and aggregates N, TP, FP, and FN counts across all grading workbooks, then derives precision, recall, and F1 metrics for CombinedHumanGrading vs. LLM AutoGrading in the 2-stage 6-example setup.

**Usage:**
```bash
python _scripts/compute/compute_rq2_grading_results.py
```

**Outputs** (written to `_Figures/data/rq2/`):
- `rq2_2stage_6examples_Metrics_CombinedHumanVsLLM.csv` — aggregated precision, recall, F1, and N/TP/FP/FN metrics across all 9 examples, per grader and element type

#### `compute_rq2_weighted_kappa.py`

**Purpose:** Computes linear weighted Cohen's kappa between human and LLM grading scores, using the aggregated confusion matrix data. Scores {0, 0.5, 1} are mapped to integers {0, 1, 2} so sklearn produces the correct 3×3 ordinal weight matrix.

**Usage:**
```bash
python _scripts/compute/compute_rq2_weighted_kappa.py
```

**Input:** `_Figures/data/rq2/rq2_confusion_matrices.csv`

**Output:** `_Figures/data/rq2/rq2_weighted_kappa_results.csv` (also printed to stdout)

#### `compute_rq2_human_grading_agreement.py`

**Purpose:** Reads the `Weighted Cohens Kappa` sheet from every `_HumanGrading_` xlsx workbook (all three configurations: 1-stage 3-shot, 2-stage 3-shot, 2-stage 6-shot) and computes the linear weighted Cohen's kappa between the two human raters for each assessment (project, approach). Reports average kappa and the percentage of assessments exceeding κ = 0.80, both per approach and aggregated across all configurations.

**Usage:**
```bash
python _scripts/compute/compute_rq2_human_grading_agreement.py
```

**Output** (written to `_Figures/data/rq2/`):
- `rq2_human_grading_agreement.csv` — one row per (approach, project) with kappa and n; aggregate rows (project = `Aggregate`) appended for each approach and for the overall total, with `pct_above_0.80` filled in for aggregate rows only

#### `compute_rq3_grader_stability.py`

**Purpose:** Reads `grading_results__generated-by-*__graded-by-*.tsv` files and computes score distributions per generator/grader/element type, per-grader stability metrics (baseline vs. comparison generator deltas), and cross-grader rankings by score-1 rate for RQ3.

**Usage:**
```bash
python _scripts/compute/compute_rq3_grader_stability.py
python _scripts/compute/compute_rq3_grader_stability.py --out-dir /custom/path
python _scripts/compute/compute_rq3_grader_stability.py --print
```

**Outputs** (written to `_Figures/data/rq3/`):
- `rq3_grading_score_distribution.csv` — score distributions for GPT/Gemini-generated state machines
- `rq3_per_grader_stability.csv` — per-grader stability metrics
- `rq3_cross_grader_rankings.csv` — cross-grader score-1 rankings for RQ3

#### `compute_element_distribution.py`

**Purpose:** Counts ground-truth model elements by type across all 9 evaluation systems, reading from the reference sheet of each `CombinedHumanGradingVsClaude4.5Sonnet` workbook. Rows labelled `"additional elements"` (FP template rows) are excluded.

**Usage:**
```bash
python _scripts/compute/compute_element_distribution.py
```

**Output:** `_Figures/data/distribution_of_state_machine_elements_across_grading_sheets.csv`
- Columns: `System, States, Transitions, Guards, Actions, Hierarchical States, Parallel Regions, History States`

### `_scripts/generate/`

#### `generate_agreement_figures.py`

**Purpose:** Generates row-normalized confusion matrix figures, both per evaluation run and aggregated across all runs. All outputs are written under `_Figures/confusion_matrices/`.

Discovers all `*CombinedHumanGrading*.xlsx` files and generates a separate set of confusion matrices **per AutoGrading LLM found**, aggregated only across files that share the same AutoGrading LLM.
- **Source files:** `*CombinedHumanGrading*.xlsx` (across all date-folders)
- **Per-file output:** `_Figures/confusion_matrices/ConfusionMatrices_CombinedHumanVs<AutoGradingLLM>_<GenerationLLM>/per_file/<Project>_<stage>_<examples>/`
- **Aggregated output:** `_Figures/confusion_matrices/ConfusionMatrices_CombinedHumanVs<AutoGradingLLM>_<GenerationLLM>/`

**How it works:** The script reads the `Weighted Cohens Kappa` sheet from matching xlsx files. It builds 3×3 confusion matrices (scores: 0, 0.5, 1) for eight scopes:

| Scope | Description |
|-------|-------------|
| Global | All elements combined |
| State | State elements only |
| Transition | Transition elements only |
| Composite State | Composite state elements only |
| Guard | Guard elements only |
| Action | Action elements only |
| History State | History state elements only |
| Region | Region elements only |

**Per-file outputs** — written to `<group_dir>/per_file/<Project>_<stage>_<examples>/`:
- `confusion_matrices.png` — combined 2×4 grid (all 8 scopes)
- `global.png`, `state.png`, … `region.png` — one figure per scope

**Aggregated outputs** — pooled across all matched files, written to `<group_dir>/`:
- prefixed with `AllExamples_CombinedHumanVs<AutoGradingLLM>_<GenerationLLM>_`

**Usage:**
```bash
# All files
python _scripts/generate/generate_agreement_figures.py

# 2-stage files only
python _scripts/generate/generate_agreement_figures.py --stage 2

# Specific date
python _scripts/generate/generate_agreement_figures.py --date 2026-03-16
```

#### `generate_paper_rq2_confusion_matrices.py`

**Purpose:** Generates publication-ready global confusion matrix figures. Aggregates all 2-stage/6-examples `CombinedHumanGrading` xlsx files per AutoGrading LLM and saves a single title-free Global figure per LLM.

**Usage:**
```bash
python _scripts/generate/generate_paper_rq2_confusion_matrices.py
```

**Outputs** (written to `_Figures/confusion_matrices/rq2_paper_figures/`):
- `fig_rq2_confusion_global_<llm>.png` — one file per AutoGrading LLM (e.g., `claude4-5sonnet`, `gpt-5-5`, `gemini3-1propreview`)

### `_scripts/checks/`

#### `check_cohens_kappa_consistency.py`

**Purpose:** Validates that the grade values stored in Sheet 2 (human rater) and Sheet 3 (LLM grader) match the values copied into Sheet 6 (`Weighted Cohens Kappa`). Discrepancies indicate copy-paste errors in the workbooks.

**Scope:** All `*_CombinedHumanGrading*.xlsx` and `*_HumanGrading_*.xlsx` workbooks (excluding templates).

**Usage:**
```bash
python _scripts/checks/check_cohens_kappa_consistency.py
```

**Output:** `_Figures/data/checks/grading_cohens_kappa_consistency_report.csv`

#### `check_interrater_consistency.py`

**Purpose:** Validates that the grade values stored in Sheet 2 (rater 1) and Sheet 3 (rater 2) match the values copied into Sheet 5 (`Inter-rater`). Discrepancies indicate copy-paste errors in the workbooks.

**Scope:** Same file set as `check_cohens_kappa_consistency.py`.

**Usage:**
```bash
python _scripts/checks/check_interrater_consistency.py
```

**Output:** `_Figures/data/checks/grading_interrater_consistency_report.csv`

### Confusion Matrix Figures

All confusion matrix figures are centralized under `_Figures/confusion_matrices/`, with one sub-folder per comparison type.

#### `_Figures/confusion_matrices/ConfusionMatrices_CombinedHumanVs<AutoGradingLLM>_<GenerationLLM>/`

One folder exists per AutoGrading LLM (e.g. `ConfusionMatrices_CombinedHumanVsClaude4.5SonnetAutoGrading_Claude4.5SonnetGeneration/`).

**Source:** Generated by `_scripts/generate/generate_agreement_figures.py` from `*CombinedHumanGrading*.xlsx` files.

**Contents:**

## Evaluation Methodology

### Human Grading Process

Generated state machines by Claude Sonnet 4.5 were evaluated by **two independent raters** using a standardized rubric (i.e., the provided `[Example Name] Evaluation - Template.xlsx`). Each rater independently assessed the generated diagrams against the model elements defined in the sample solution without consulting one another, to reduce evaluation bias. After evaluation, inter-rater agreement was computed using **weighted Cohen's κ**. When the agreement score was **κ ≥ 0.800**, the evaluations were considered sufficiently consistent; otherwise, the evaluators jointly reconciled disagreements through discussion to produce a single agreed-upon final evaluation.

### LLM-Based Grading (Automatic Grading)

The automated assessment module uses a **zero-shot, single-prompt** configuration. Each LLM grader evaluated a submitted Mermaid state machine using four inputs:  
1. the original system description (treated as the sole source of truth for expected behavior),  
2. the structured evaluation rubric in CSV format (derived from `[Example Name] Evaluation - Template.xlsx`, without computed metrics), which serves only as a rubric template defining expected coverage,  
3. the ground truth Mermaid diagram (used solely to interpret how the rubric maps to the system description, not to force literal similarity), and  
4. the generated Mermaid diagram to be graded (assessed semantically against the intended behavior).  

The LLM preserved the rubric's CSV structure exactly, modifying only the assessment and justification columns. It assigned scores (0, 0.5, or 1) to each rubric element and generated concise, row-specific justifications, prioritizing semantic correctness over syntactic similarity. The temperature was set to **0** for assessment to ensure better reproducible results.

For generation, temperature was set to **0.01** for the first-stage and single-stage prompts (to encourage syntactic consistency) and **0.3** for the second-stage refinement prompt (to allow greater structural flexibility).

## Grading Scheme

The grading rubric evaluates multiple criteria across **states, transitions, guards, actions, composite states, regions, and history states**. Each element is scored using the following scale:

| Value | Definition |
|-------|-----------|
| **1** | Correct — matches expected behaviour |
| **0.5** | Partially correct — minor issue or incomplete |
| **0** | Incorrect or missing |
| **Additional** | Extra (unexpected) elements |

### Additional Elements (False Positives)

Additional elements are treated as **false positives** in the grading metrics. They appear in rows where column B of the grading sheet contains `"additional element"`.

- The column-C value of each additional-element row is an **integer count** of spurious elements of that type (e.g. 0, 1, 2, …). These counts are summed across all additional-element rows of a given type to produce **FP**.
- They **lower precision** and therefore **reduce F1-score**.

In the confusion matrices they are mapped by comparing the raw human and LLM false-positive counts per element type:

| Condition | Confusion matrix cell |
|-----------|----------------------|
| Both graders found **no** extra elements (`human=0`, `llm=0`) | `[0, 0]` — agreement at score 0 |
| LLM hallucinated more extras than human (`llm > human`) | `[0, 2]` — LLM over-generates |
| Human found more extras than LLM (`human > llm`) | `[2, 0]` — human found more FPs |
| Both found extras | `[2, 2]` += `min(human, llm)` — shared agreement on extras |

> **Note:** A global consistency section (evaluating formatting and naming conventions) was initially included but removed as too subjective. The evaluation now focuses on semantic correctness and functional completeness rather than stylistic conventions.

The final output is a completed grading sheet where each element is scored and justified concisely, ensuring alignment with expected system behavior.

## Performance Metrics Explanation

A **weighted scoring** scheme is used so that partial matches (score 0.5) are credited less than full matches (score 1).

### How metrics are gathered

The values reported in the output CSVs (`rq1_generation_quality_by_example.csv`, `rq1_generation_quality_summary.csv`, and `rq2_2stage_6examples_Metrics_CombinedHumanVsLLM.csv`) are **not** taken from the pre-computed Metrics sheet inside the individual Excel workbooks. Instead, the compute scripts read the raw grading scores directly from the individual grading sheets (column C of the Human Grading sheet for RQ1 and RQ2 human grading; column C of the LLM Grading sheet for RQ2 LLM grading) and compute all aggregated metrics in code using the formulas below.

For each element type, the rows of a grading sheet are split into two zones by detecting the first row where column B contains `"additional element"`. Rows before that point are **expected elements** (ground truth); rows from that point onward are **additional elements** (false positives).

| Symbol | Definition | How it is computed |
|--------|-----------|-------------------|
| **S** | Weighted true positives | Sum of column-C scores for all expected rows of that type (each score s_i ∈ {0, 0.5, 1}) |
| **TP** | Binary true-positive count | Count of expected rows with score > 0 |
| **FP** | False-positive count | Sum of column-C integer counts from additional-element rows of that type |
| **FN** | False-negative count | Count of expected rows with score = 0 |
| **N** | Ground truth count | TP + FN (total expected elements of that type) |

The metrics are then:

| Metric | Formula |
|--------|---------|
| **Precision** | S / (TP + FP) |
| **Recall** | S / (TP + FN) |
| **F1-Score** | 2 × Precision × Recall / (Precision + Recall) |

**Interpretation:**
- **High Precision** → High average correctness score per predicted element, and few spurious extras
- **High Recall** → High average correctness score per expected element, and few missed elements
- **High F1-Score** → Well-balanced correctness and completeness

> [!WARNING]
> **Known bug — Metrics sheets inside all Excel files**
>
> **Note on the individual Metrics sheets:** Each Excel workbook contains a pre-computed Metrics sheet that was used as a development aid during data collection. That sheet contains a bug: it does not distinguish between **S** (sum of partial scores) and **TP** (binary count of matched elements), treating them as equal (i.e., it uses S in place of TP in the denominator). As a result, the Precision and Recall values displayed in the individual Metrics sheets are incorrect and differ from the values reported in the final output CSVs. The individual Metrics sheets are **not** the source of any reported results.
